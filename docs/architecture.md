# Phase 1: Technical Architecture & Governance

## 1. End-to-End System Flow

The Agentic dbt Librarian is an event-driven pipeline that connects GitHub, n8n, an LLM, and your dbt project. Here is the precise flow:

```
┌─────────────────────────────────────────────────────────────────────┐
│                         TRIGGER LAYER                               │
│                                                                     │
│  Developer pushes a commit to GitHub containing changes to a        │
│  *.sql file in the /models directory of the dbt project.            │
│                                                                     │
│  GitHub fires a "push" Webhook Event → n8n Webhook Node             │
└────────────────────────────┬────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│                       CONTEXT ASSEMBLY LAYER                        │
│                                                                     │
│  n8n fetches three data sources in parallel:                        │
│                                                                     │
│  1. The raw .sql file content (from GitHub API / repo clone)        │
│  2. manifest.json  → node graph, model dependencies, existing docs  │
│  3. catalog.json   → column types, warehouse-level metadata         │
│                                                                     │
│  These are assembled into a structured "Prompt Context" object.     │
└────────────────────────────┬────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│                       INTELLIGENCE LAYER                            │
│                                                                     │
│  The AI Agent Node receives:                                        │
│    • System Prompt (Impact Prompting - see Phase 4)                 │
│    • The SQL model code                                             │
│    • Upstream model descriptions (from manifest.json lineage)       │
│    • Column data types (from catalog.json)                          │
│                                                                     │
│  The LLM returns: a valid dbt schema.yml YAML block                 │
└────────────────────────────┬────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│                      VALIDATION LAYER                               │
│                                                                     │
│  Response Parser Node:                                              │
│    • Strips markdown code fences (```yaml ... ```)                  │
│    • Validates the YAML can be parsed without errors                │
│    • Checks that required keys exist (version, models, columns)     │
│    • If invalid → sends to an error Slack/email notification        │
└────────────────────────────┬────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│                       DELIVERY LAYER                                │
│                                                                     │
│  GitHub API Node:                                                   │
│    • Creates a new branch: docs/ai-schema-{model_name}-{timestamp}  │
│    • Commits the schema.yml file to the branch                      │
│    • Opens a Pull Request with a pre-filled description explaining  │
│      what the AI generated and why                                  │
│                                                                     │
│  A human Data Engineer reviews and merges the PR.                   │
└─────────────────────────────────────────────────────────────────────┘
```

## 2. Metadata Strategy: Using manifest.json and catalog.json

### Why Both Files Are Needed

The AI Agent cannot understand *business context* from SQL alone. It needs the **lineage graph** and **physical schema** to write descriptions that are accurate and non-redundant.

### manifest.json — The Knowledge Graph

`manifest.json` is generated by `dbt compile` or `dbt run`. It contains:

| Key Path | What It Tells the Agent |
|----------|------------------------|
| `nodes.{model}.depends_on.nodes` | Which upstream models this model reads from (lineage) |
| `nodes.{model}.description` | Existing human-written description (to avoid overwriting) |
| `nodes.{model}.columns.{col}.description` | Existing column descriptions |
| `nodes.{model}.raw_code` | The raw SQL for the model |
| `sources.{source}.description` | Descriptions of raw source tables |

**How the Agent Uses It:**
1. Extract the `depends_on.nodes` list for the model being analyzed.
2. Fetch the `description` field for each upstream model.
3. Inject these descriptions as "Parent Context" into the prompt — so the AI knows "this model is downstream of `fct_orders`, which tracks completed revenue transactions."
4. Skip generating descriptions for columns that already have non-empty descriptions.

### catalog.json — The Physical Schema

`catalog.json` is generated by `dbt docs generate`. It contains:

| Key Path | What It Tells the Agent |
|----------|------------------------|
| `nodes.{model}.columns.{col}.type` | The actual Snowflake data type (VARCHAR, NUMBER, TIMESTAMP_NTZ, etc.) |
| `nodes.{model}.columns.{col}.index` | Column ordinal position |
| `nodes.{model}.stats` | Row count, bytes, clustering keys |

**How the Agent Uses It:**
1. Enrich each column with its data type → helps AI infer semantics (e.g., `TIMESTAMP_NTZ` → "event time", `NUMBER(38,2)` → "monetary amount").
2. Identify columns in the warehouse that are missing from `schema.yml` entirely.
3. Use stats (row count) to add "scale" context to descriptions.

### Lineage-Aware Prompt Construction (Pseudocode)

```python
def build_agent_context(model_name, manifest, catalog):
    model_node = manifest["nodes"][f"model.{project}.{model_name}"]
    
    # Get SQL
    sql = model_node["raw_code"]
    
    # Get upstream context
    parent_descriptions = []
    for parent_id in model_node["depends_on"]["nodes"]:
        parent = manifest["nodes"].get(parent_id, {})
        if parent.get("description"):
            parent_descriptions.append(
                f"- {parent['name']}: {parent['description']}"
            )
    
    # Get column types from catalog
    catalog_cols = catalog["nodes"].get(
        f"model.{project}.{model_name}", {}
    ).get("columns", {})
    
    col_types = {
        col_name: col_data["type"] 
        for col_name, col_data in catalog_cols.items()
    }
    
    return {
        "sql": sql,
        "parent_context": "\n".join(parent_descriptions),
        "column_types": col_types,
        "existing_descriptions": model_node.get("columns", {})
    }
```

## 3. Governance Controls

| Control | Mechanism |
|---------|-----------|
| **Human-in-the-loop** | All AI output goes through a GitHub PR — no auto-merge |
| **Description locking** | Existing non-empty descriptions are passed to the AI with instruction to preserve them |
| **Schema validation** | YAML parser + jsonschema validation before PR creation |
| **Audit trail** | Git history records every AI-generated description change |
| **Gap reporting** | `gap_finder.py` generates weekly `DOCS_AUDIT.md` to track coverage |
