The Problem

Why Documentation Debt Kills Data Teams

Every dbt project starts clean. Then reality hits — deadlines, sprint pressure, and the YAML never gets written.

🔴

60% of columns ship undocumented

Business users can't tell if net_revenue is before or after returns. They guess wrong. Revenue dashboards silently break.

How It Works

From Git Push to Pull Request in 30 Seconds

A fully event-driven pipeline that turns code changes into documentation automatically.

1

Developer pushes .sql to GitHub

Any new or modified model file in models/ triggers the pipeline via GitHub webhook.

WEBHOOK
2

n8n fetches SQL + manifest.json in parallel

The orchestration engine grabs the raw model SQL and the compiled dbt manifest for upstream lineage context.

PARALLEL FETCH
3

Lineage-aware context assembly

Merges SQL code, upstream model descriptions, existing column docs (to preserve), materialization type, and column data types into a structured prompt.

CONTEXT ENGINE
4

Gemini AI generates schema.yml via Impact Prompting

Every description answers: What is it? (technical) · Why does it exist? (business) · What breaks if it's wrong? (risk)

AI AGENT
5

YAML validated → Branch created → PR opened

The output is parsed, validated for correct YAML structure, committed to a new branch, and a Pull Request is opened — all automatically.

Enterprise Governance

Built for Production, Not Just Demos

👤

Human-in-the-Loop

Every AI output goes through a GitHub Pull Request. No auto-merge — a data engineer always reviews before it hits main.

🔒

Description Locking

Existing non-empty descriptions are passed to the AI with explicit instructions to preserve them. No accidental overwrites.

✅

YAML Validation

AI output is parsed and validated for correct YAML structure before any branch or PR is created. Bad output = error response, no PR.

📋

Git Audit Trail

Every AI-generated description change is recorded in git history. Full traceability for compliance and data governance.

🚨

CI Alerting

GitHub Actions runs the Gap Finder on every push. CRITICAL gaps automatically open a GitHub Issue for the team.
